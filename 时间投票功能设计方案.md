# 时间投票功能设计方案

## 一、可行性分析

### ✅ 完全可以实现

这个功能在技术上没有任何障碍，核心逻辑清晰，小程序端可以完整承载。

**核心要素：**
- 每个人提交可用时间段（多日多时段）
- 系统计算交集并排序
- 考虑"让步意愿"作为权重调整
- 投票确认最终时间

## 二、核心算法设计

### 1. 数据结构

```javascript
// 每个参与者提交的数据
{
  openId: "xxx",
  name: "张三",
  avatar: "url",
  availability: {
    "2026-05-12": ["12:00", "18:00", "19:00"],  // 具体可用时间点
    "2026-05-13": ["18:00", "19:00", "20:00"],
    "2026-05-14": ["12:00", "13:00"]
  },
  flexibility: 2,  // 让步意愿等级：0=绝不，1=有点难，2=可以商量，3=完全OK
  preferredSlots: ["2026-05-12_18:00", "2026-05-13_19:00"]  // 最偏好时段
}
```

### 2. 共识算法（核心）

```javascript
/**
 * 计算最佳公共时间
 * @param {Array} participants 所有参与者数据
 * @param {Number} minRequired 最低需要多少人（默认全部）
 * @returns {Array} 按推荐度排序的时间段
 */
function findConsensusTime(participants, minRequired = null) {
  const required = minRequired || participants.length;
  const slotScores = {};  // 每个时间段的得分
  const slotPeople = {};  // 每个时间段可用的人

  // 第一步：统计每个时间段有多少人可用
  participants.forEach(p => {
    const scoreWeight = p.flexibility + 1;  // 让步意愿高的权重更高

    Object.entries(p.availability).forEach(([date, times]) => {
      times.forEach(time => {
        const key = `${date}_${time}`;
        if (!slotPeople[key]) {
          slotPeople[key] = [];
          slotScores[key] = 0;
        }
        slotPeople[key].push(p.openId);
        slotScores[key] += scoreWeight;  // 累加权重分
      });
    });
  });

  // 第二步：计算每个时间段的"共识度"
  const results = Object.keys(slotPeople).map(key => {
    const [date, time] = key.split('_');
    const availableCount = slotPeople[key].length;
    const totalScore = slotScores[key];

    // 共识度 = 可用人数比例 × 权重分 × 偏好加成
    const coverage = availableCount / participants.length;
    const preferenceBonus = participants.filter(p =>
      p.preferredSlots.includes(key)
    ).length * 2;  // 偏好时段额外加分

    const consensusScore = coverage * 100 + totalScore * 0.5 + preferenceBonus * 10;

    return {
      date,
      time,
      availableCount,
      totalParticipants: participants.length,
      coverage: Math.round(coverage * 100),
      consensusScore: Math.round(consensusScore),
      availablePeople: slotPeople[key],
      missingPeople: participants
        .filter(p => !slotPeople[key].includes(p.openId))
        .map(p => ({ name: p.name, flexibility: p.flexibility }))
    };
  });

  // 第三步：排序
  // 优先：覆盖率高 → 共识分高 → 时间早
  return results
    .filter(r => r.availableCount >= required)
    .sort((a, b) => {
      if (b.coverage !== a.coverage) return b.coverage - a.coverage;
      if (b.consensusScore !== a.consensusScore) return b.consensusScore - a.consensusScore;
      return new Date(a.date + 'T' + a.time) - new Date(b.date + 'T' + b.time);
    });
}
```

### 3. 让步逻辑

当没有100%覆盖的时间段时，系统会：

```javascript
/**
 * 智能推荐（包含让步建议）
 */
function getSmartRecommendation(participants) {
  // 先尝试找全员共识
  let bestSlots = findConsensusTime(participants, participants.length);

  if (bestSlots.length === 0) {
    // 没有全员共识，找最多人共识
    for (let i = participants.length - 1; i >= 2; i--) {
      bestSlots = findConsensusTime(participants, i);
      if (bestSlots.length > 0) {
        // 标注这是"多数人共识"，并给出让渡建议
        return {
          type: 'majority',
          requiredPeople: i,
          slots: bestSlots.slice(0, 3),  // 推荐前3个
          suggestion: generateSuggestion(participants, bestSlots[0])
        };
      }
    }
  }

  return {
    type: 'consensus',
    slots: bestSlots.slice(0, 3),
    suggestion: null
  };
}

/**
 * 生成让渡建议话术
 */
function generateSuggestion(participants, bestSlot) {
  const flexibles = bestSlot.missingPeople.filter(p => p.flexibility >= 2);
  const stubborn = bestSlot.missingPeople.filter(p => p.flexibility <= 1);

  if (flexibles.length > 0 && stubborn.length === 0) {
    const names = flexibles.map(p => p.name).join('、');
    return `${names}表示可以商量，建议选择这个时段`;
  } else if (stubborn.length > 0) {
    const names = stubborn.map(p => p.name).join('、');
    return `${names}时间确实不方便，可能需要单独协调或分两场`;
  }
  return null;
}
```

### 为什么不放在现有模式里？

| 方案 | 优点 | 缺点 |
|------|------|------|
| **独立入口（推荐）** | 逻辑清晰，复用性高，可作为前置步骤 | 需要新增页面 |
| 放到模式A | 流程顺畅，选完时间直接选店 | 强制捆绑，不够灵活 |
| 放到模式B | 符合"大家来定"的理念 | 模式B已有口味投票，会混淆 |
| 放到"约饭" | 约饭本身就需要协调时间 | 聚餐也需要这个功能 |

## 九、替代轻量方案（如果工期紧）

如果暂时不想做大功能，可以先做一个轻量版：

在现有"聚餐/约饭"创建流程中，增加一步：
1. 创建者选择3-5个候选时间段
2. 分享到群里，大家用小程序投票（类似现有的投票机制复用）
3. 票数最高的时间段获胜
4. 自动填入活动时间

这个方案复用现有投票页面（`/pages/vote/vote`），只需增加"时间段选项"类型，工作量约2天。
